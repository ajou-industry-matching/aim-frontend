# SEO 대응을 위한 Next.js 전환 구조 설계

## 1. 목적

현재 프론트엔드는 `Vite + React` 기반 구조와 `Next.js App Router` 구조가 일부 혼재되어 있다.  
이 문서는 SEO가 필요한 페이지를 적절한 렌더링 방식으로 제공하기 위해, 현재 구조를 기준으로 Next.js 전환 방향과 적용 범위를 정리한다.  
현재 전제는 `Firebase Hosting 유지`이며, 따라서 전환의 기본 방향은 `SSR 전면 도입`이 아니라 `SSG 중심 설계`다. Firebase Hosting은 `out/` 정적 산출물을 직접 배포하는 구조를 기준으로 한다.

## 2. 현재 구조 분석

### 2.1 확인된 현황

- 실행 스크립트는 `next dev`, `next build`, 정적 산출물 미리보기 기준으로 정리되어 있다.
- `src/app/layout.tsx`, `src/app/page.tsx`, `src/app/(auth)/login/page.tsx`가 존재해 App Router 기반 진입점이 일부 구성되어 있다.
- 당시에는 `src/main.tsx`, `vite.config.ts`, `index.html`, `public/index.html`이 함께 존재해 Vite 자산이 공존했다.
- 현재 라우팅은 복잡하지 않으며 `react-router` 의존성은 없다.
- 현재 공개 페이지로 확인되는 것은 `/login` 중심이며, `src/app/page.tsx`는 `/login`으로 리다이렉트한다.
- UI 컴포넌트 중 일부는 `useState`, `useEffect`, `document`를 사용하므로 Client Component 경계 설정이 중요하다.
- Firebase 설정은 당시 `process.env.NEXT_PUBLIC_*`와 `import.meta.env.VITE_*`를 동시에 읽고 있었다.

### 2.2 구조상 해석

- 이 프로젝트는 "Vite에서 Next.js로 완전히 넘어가는 초기 단계"로 보는 것이 맞다.
- 따라서 이번 이슈의 핵심은 Next.js 신규 도입이 아니라, 혼재 상태를 정리하면서 SEO 기준에 맞는 렌더링 정책을 수립하는 것이다.

## 3. 페이지 분류 원칙

### 3.1 SEO 대상 페이지

다음 조건 중 하나라도 만족하면 정적 생성 가능한지 먼저 검토한다.

- 로그인 전에도 접근 가능한 공개 페이지다.
- 검색엔진이 제목, 설명, 본문을 수집해야 한다.
- 공유 시 메타 태그(OG/Twitter), canonical, 구조화 데이터가 중요하다.
- URL별로 독립적인 콘텐츠가 존재한다.

권장 렌더링 전략:

- 기본값은 `SSG`
- 데이터 갱신 주기가 있으면 `ISR`
- `SSR`은 Firebase Hosting 유지 조건과 충돌 가능성이 있으므로 예외적으로만 검토

예상 후보:

- 서비스 소개 페이지
- 학교/기업 대상 소개 페이지
- 공지사항 목록/상세
- 공개 포트폴리오 목록/상세
- 모집/프로그램 소개 페이지

### 3.2 CSR 유지 가능 페이지

다음 조건이면 CSR 또는 Client Component 중심 구성을 유지한다.

- 로그인 이후에만 접근 가능한 페이지다.
- 검색엔진 노출 가치보다 사용자 상호작용이 더 중요하다.
- 대시보드, 입력 폼, 편집 화면처럼 인증/실시간 상태 의존이 크다.

예상 후보:

- 로그인 페이지
- 마이페이지
- 포트폴리오 작성/수정 화면
- 관리자 대시보드
- 기업/학생 전용 내부 기능

## 4. 현재 코드 기준 1차 분류

현재 저장소에서 실제로 확인되는 페이지 기준으로는 아래와 같이 분류할 수 있다.

| 경로 | 현재 역할 | 권장 방식 | 근거 |
| --- | --- | --- | --- |
| `/` | `/login` 리다이렉트 | CSR 리다이렉트 유지 | 정적 export 환경에서 진입 제어 역할 |
| `/login` | 로그인 진입 | CSR 유지 | 인증/탭 전환/입력 상태 중심, 검색 노출 가치 낮음 |

추가로 향후 공개 페이지가 생기면 SEO 대상군으로 별도 분리해 App Router 하위에 배치한다.

## 5. 전환 시 영향 범위

### 5.1 라우팅

현재 영향:

- `src/app` 기준 라우팅은 이미 Next.js 방식으로 시작되어 있다.
- `react-router` 전환 비용은 사실상 없다.

대응 방향:

- 신규 페이지는 모두 `src/app/**/page.tsx` 기준으로 정의한다.
- 공개 페이지와 인증 후 페이지를 route group으로 분리한다.
- 공개 페이지는 정적 생성 가능한 구조를 우선한다.

권장 예시:

```text
src/app
  /(public)
    page.tsx
    about/page.tsx
    notice/page.tsx
    notice/[id]/page.tsx
    portfolio/[id]/page.tsx
  /(auth)
    login/page.tsx
  /(dashboard)
    dashboard/page.tsx
    portfolio/edit/page.tsx
```

### 5.2 스타일

현재 영향:

- 전역 스타일은 `src/index.css`에 토큰과 베이스 스타일이 모여 있다.
- Next 전역 스타일은 `src/app/globals.css`에서 `src/index.css`를 import 하는 형태다.
- 일부 화면은 CSS Module(`login.module.css`)을 함께 사용한다.

대응 방향:

- 디자인 토큰과 reset은 `src/app/globals.css` 기준으로 유지한다.
- 페이지/컴포넌트 단위 스타일은 CSS Module 또는 Tailwind utility 조합으로 유지한다.
- Vite 전용 진입점(`src/main.tsx` 기반 import 체인)에 의존하는 스타일 참조는 제거한다.

### 5.3 환경 변수

현재 영향:

- Firebase 설정이 `NEXT_PUBLIC_*`와 `VITE_*`를 모두 읽는다.
- 혼재 상태에서는 동작하지만 장기적으로는 유지 비용이 커진다.

대응 방향:

- 브라우저 노출이 필요한 값은 `NEXT_PUBLIC_*`로 통일한다.
- 서버 전용 값은 Next 서버 환경 변수로만 사용한다.
- `import.meta.env` 의존성은 제거한다.

### 5.4 클라이언트 전용 로직

현재 영향:

- `LoginPage`, `Navigation`, 일부 입력/드롭다운 컴포넌트는 `use client` 또는 DOM API 사용이 필요하다.

대응 방향:

- 페이지 전체를 무조건 Client Component로 만들지 않는다.
- 상호작용이 필요한 섹션만 Client Component로 분리한다.
- SEO 대상 페이지는 가능하면 Server Component를 기본으로 두고, 필요한 위젯만 client island로 분리한다.

### 5.5 상태 관리 및 폼 스택

2단계 기준으로 채택 방향은 아래와 같다.

- `Next.js App Router`: 라우팅, 정적 생성, 기본 데이터 패칭
- `react-hook-form`: 입력 폼 상태 관리
- `zod`: 폼/DTO 검증 및 스키마 기반 타입 추론
- `zustand`: 클라이언트 전역 상태 관리

역할 분담 원칙:

- 서버 데이터와 SEO 대응은 Next.js 기본 기능 중 `SSG`를 우선 사용한다.
- 로그인, 회원가입, 프로필 수정, 포트폴리오 작성 같은 폼은 `react-hook-form + zod` 조합을 기본으로 사용한다.
- 모달, 필터, 다단계 작성 상태, 사용자 세션 기반 UI 상태처럼 여러 컴포넌트에서 공유되는 클라이언트 상태는 `zustand`로 관리한다.
- 단순 로컬 상태는 여전히 `useState`로 처리한다.

### 5.6 확정된 결정사항

이번 논의에서 아래 항목을 확정한다.

- 폼 스택 도입 시점:
  - 인증/회원가입 화면이 실제로 구현되는 시점부터 `react-hook-form + zod`를 도입한다.
- `zustand` 적용 범위:
  - 전역 store를 선제적으로 만들지 않고, 여러 컴포넌트에서 공유되는 클라이언트 상태가 생길 때 최소 범위로 도입한다.
- Firebase Hosting 전략:
  - Firebase Hosting은 유지한다.
  - `out/` 정적 산출물을 직접 배포한다.
- 공개 페이지 렌더링 전략:
  - 기본은 `SSG`
  - 공지사항 계열만 추후 `ISR` 가능성을 열어둔다.
  - 그 외 공개 페이지는 `SSG`를 우선 기준으로 삼는다.

## 6. 권장 전환 전략

### 6.1 전략 요약

이번 전환은 "전체 페이지를 SSR로 바꾸는 작업"이 아니다.  
Firebase Hosting 유지가 우선 조건이므로, "SEO가 필요한 공개 페이지는 `SSG 중심`으로 설계하고, 내부 기능 페이지는 CSR 중심으로 분리"하는 작업으로 정의한다.

핵심 원칙:

- 공개 SEO 페이지: `SSG` 우선
- 데이터 갱신이 필요한 공개 페이지: `ISR` 검토
- 로그인 이후 내부 기능 페이지: `CSR` 유지
- `SSR`: 비용/배포 복잡도 증가가 정당화되는 경우에만 예외적으로 검토

### 6.2 단계별 진행

#### 1단계. 구조 정리

- `src/app`를 단일 라우팅 기준으로 확정한다.
- Vite 잔존 진입점(`src/main.tsx`, `index.html`)과 환경 변수 의존성을 제거한다.
- Firebase Hosting은 `out/` 정적 산출물을 직접 배포하는 기준으로 정리한다.
- README와 실제 기술 스택 차이를 맞춘다.

#### 2단계. 페이지 인벤토리 작성

- 전체 화면 목록을 작성한다.
- 각 페이지를 `SEO 필요 / 불필요`, `공개 / 비공개`, `SSR / SSG / ISR / CSR`로 분류한다.

#### 3단계. 공개 페이지 우선 전환

- 소개, 공지, 공개 포트폴리오 등 검색 유입 대상 페이지부터 Next App Router 규칙으로 확정한다.
- 각 페이지에 `metadata`, OG, canonical 정책을 붙인다.
- Firebase Hosting에서 정적으로 배포 가능한 데이터 구조인지 먼저 확인한다.

#### 4단계. 클라이언트 경계 최적화

- 공개 페이지에서 `use client` 범위를 최소화한다.
- 인증 후 화면은 기존 React 상호작용 구조를 유지하되 Next route 하위로 옮긴다.
- 폼은 `react-hook-form + zod`, 전역 UI 상태는 `zustand` 기준으로 공통 패턴을 정리한다.

#### 5단계. 환경 변수와 배포 기준 정리

- `NEXT_PUBLIC_*` 기준으로 환경 변수를 통일한다.
- Firebase Hosting 기준에서 `out/` 정적 산출물 배포 경로를 검증한다.
- `ISR` 필요 여부는 실제 공개 페이지 요구사항이 생길 때 별도 검토한다.

## 8. 페이지 인벤토리 및 1차 분류

### 8.1 현재 코드에서 확인된 페이지

현재 `src/app`와 `src/views` 기준으로 실제 확인되는 페이지는 많지 않다.  
아래 항목은 저장소에 존재하는 코드 기준으로 확인 가능한 페이지들이다.

| 구분 | 경로 | 현재 상태 | 공개 여부 | SEO 필요 | 권장 렌더링 | 비고 |
| --- | --- | --- | --- | --- | --- | --- |
| 확정 | `/` | `/login`으로 리다이렉트 | 공개 | 낮음 | 서버 리다이렉트 | [src/app/page.tsx] 기준 |
| 확정 | `/login` | 로그인 진입 페이지 | 공개 | 낮음 | CSR | `src/app/(auth)/login/page.tsx`, 탭 전환/입력 상태/인증 처리 중심 |
| 참고 | `HomePage` | Storybook/샘플 성격 view | 미정 | 낮음 | 미정 | 실제 라우트 미연결, SEO 대상 아님 |

### 8.2 UI 문맥상 확인되는 예정 페이지

아래 항목은 실제 route 파일이 존재하지는 않지만, 네비게이션/스토리/문구 기준으로 조만간 필요할 가능성이 높은 화면이다.  
이 항목들은 구현 전에 기획과 함께 공개 여부와 SEO 필요성을 확정해야 한다.

| 구분 | 예상 경로 | 근거 | 공개 여부(예상) | SEO 필요(예상) | 권장 렌더링 |
| --- | --- | --- | --- | --- | --- |
| 예정 | `/portfolio` 또는 `/portfolio/[id]` | 로그인 화면/네비게이션에 `포트폴리오` 노출 | 공개 가능성 높음 | 높음 | SSG 우선, 필요 시 ISR |
| 예정 | `/about` | 로그인 화면/네비게이션에 `소개` 노출 | 공개 | 높음 | SSG |
| 예정 | `/notice` | 로그인 화면/네비게이션에 `공지사항` 노출 | 공개 | 높음 | SSG 또는 ISR |
| 예정 | `/notice/[id]` | 공지사항 상세 필요 | 공개 | 높음 | SSG 우선, 필요 시 ISR |
| 예정 | `/signup` | `회원가입으로 이동하세요` 문구 존재 | 공개 | 낮음 | CSR |
| 예정 | `/profile` | 네비게이션 드롭다운의 `내 프로필` | 비공개 | 낮음 | CSR |
| 예정 | `/my-portfolio` 또는 `/portfolio/edit` | 드롭다운의 `내 포트폴리오` | 비공개 | 낮음 | CSR |
| 예정 | `/dashboard` | 관리자 대시보드 문구 존재 | 비공개 | 낮음 | CSR |

### 8.3 분류 원칙 적용 메모

- 검색 유입을 노리는 소개/공지/공개 포트폴리오는 SEO 대상군으로 본다.
- 인증 후 사용자 데이터 중심 화면은 CSR 유지가 자연스럽다.
- 로그인 페이지는 공개 페이지이지만 검색 유입 가치가 낮고 상호작용 비중이 높아 CSR이 적합하다.
- `/`는 독립 콘텐츠가 아니라 진입 분기 역할이므로 서버 리다이렉트가 가장 단순하다.

### 8.4 페이지 유형별 권장 스택

| 페이지 유형 | 예시 | 권장 렌더링 | 권장 스택 |
| --- | --- | --- | --- |
| 공개 SEO 페이지 | 소개, 공지사항, 공개 포트폴리오 | SSG 우선, 필요 시 ISR | Next.js Server Component, `generateMetadata` |
| 인증 진입 폼 | 로그인, 회원가입 | CSR | `react-hook-form`, `zod` |
| 인증 후 입력 화면 | 포트폴리오 작성/수정, 프로필 수정 | CSR | `react-hook-form`, `zod`, 필요 시 `zustand` |
| 전역 UI 상태 화면 | 대시보드 필터, 모달, 작성 단계 | CSR | `zustand` |
| 단순 상호작용 컴포넌트 | 탭, 토글, 드롭다운 | Client Component | `useState` 우선, 필요 시 `zustand` |

### 8.5 2단계 결과 요약

현재 기준에서 확정 가능한 SEO 대상 페이지는 아직 코드상 존재하지 않는다.  
다만 서비스 문맥상 `소개`, `공지사항`, `공개 포트폴리오` 계열은 SEO 우선 페이지로 보는 것이 타당하다.

따라서 다음 구현 단계에서는 아래 순서로 라우트를 구체화하는 것이 좋다.

1. 공개 라우트 후보를 실제 경로 단위로 확정한다.
2. 각 경로에 대해 `SSG / ISR / CSR`를 우선 기준으로 확정한다.
3. 현재는 `src/app/(auth)/login/page.tsx`를 기준으로 route group을 시작하고, 이후 `src/app/(public)`와 `src/app/(dashboard)`를 확장한다.
4. 인증/입력 화면의 공통 폼 패턴을 인증/회원가입 구현 시점에 `react-hook-form + zod`로 정리한다.
5. 전역 UI 상태 후보를 식별해 `zustand` store 범위를 최소화한다.

## 9. Firebase Hosting 유지 기준 설계 원칙

Firebase Hosting 유지가 우선 조건이므로, 배포 전략은 아래 원칙을 따른다.

- SEO 대상 공개 페이지는 정적 생성 가능한 형태로 우선 설계한다.
- `SSR` 전용 설계는 기본 전제에서 제외한다.
- 데이터 최신성이 필요한 공개 페이지는 먼저 `ISR 없이도 운영 가능한지` 검토한다.
- 공개 콘텐츠가 빌드 시점 생성으로 감당되지 않을 때만 `ISR` 또는 별도 배포 전략을 검토한다.
- `/index.html` SPA rewrite와 fallback 파일에는 더 이상 의존하지 않는다.

현재 기준 추천 방향:

- 소개/공지/정적 공개 콘텐츠: `SSG`
- 로그인/회원가입/내부 대시보드: `CSR`
- 공개 포트폴리오 상세: 우선 `SSG` 가능성 검토
- 공지사항 계열: 필요 시 `ISR` 논의 가능

## 10. Vite 제거 진행 현황

완전 제거 작업은 대부분 진행되었고, 현재 상태는 아래와 같다.

- 제거 완료:
  - `vite` 직접 의존성
  - `vitest`, `playwright` 직접 의존성
  - `src/main.tsx`
  - `index.html`
  - `vite.config.ts`
  - `src/firebase.js`
  - `src/app/index.tsx`
  - `src/app/index.css`
  - `import.meta.env` 의존성
- 전환 완료:
  - Storybook framework를 `@storybook/nextjs`로 변경
  - Storybook 핵심 패키지 버전을 `10.3.3`으로 정렬
즉, 로컬 개발과 Storybook 기준으로는 Vite 제거가 사실상 완료되었고,  
배포 기준으로도 Firebase Hosting을 유지하면서 `Next static export + SSG 중심` 구조로 정리했다.
