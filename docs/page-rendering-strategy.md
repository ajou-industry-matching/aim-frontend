# 페이지 렌더링 전략 정리

이 문서는 AIM 프론트엔드의 페이지별 렌더링 기준을 정리한다.  
현재 전제는 다음과 같다.

- `Firebase Hosting` 유지가 우선 조건이다.
- 공개 SEO 페이지는 `SSG`를 기본 전략으로 삼는다.
- `ISR`은 공지사항 계열처럼 갱신 주기가 명확한 공개 페이지에 한해 추후 검토한다.
- 로그인 이후 내부 기능 페이지는 `CSR`을 유지한다.
- `SSR`은 현재 기본 전략에서 제외한다.

## 1. 확정된 운영 원칙

### 1.1 공개 SEO 페이지

- 기본 전략: `SSG`
- 대상 예시:
  - 소개 페이지
  - 학생/기업/연구실 개인 페이지
  - 공지사항 목록
  - 공지사항 상세
  - 공개 포트폴리오 소개/목록

### 1.2 인증 후 내부 페이지

- 기본 전략: `CSR`
- 대상 예시:
  - 로그인
  - 회원가입
  - 프로필 수정
  - 포트폴리오 작성/수정
  - 관리자 대시보드

### 1.3 `ISR` 허용 범위

- 현재는 전면 허용하지 않는다.
- `공지사항` 계열만 추후 필요 시 검토한다.
- 공개 포트폴리오 상세는 우선 `SSG` 가능성을 먼저 본다.

## 2. 페이지 유형별 기준표

| 페이지 유형 | 예시 경로 | 공개 여부 | 기본 전략 | 비고 |
| --- | --- | --- | --- | --- |
| 진입 분기 페이지 | `/` | 공개 | CSR 리다이렉트 | 정적 export 환경에서 `/login`으로 분기 |
| 인증 진입 페이지 | `/login`, `/signup` | 공개 | CSR | 상호작용 중심, SEO 우선도 낮음 |
| 정적 공개 페이지 | `/about` | 공개 | SSG | 소개/안내형 콘텐츠 |
| 공지 목록 페이지 | `/notice` | 공개 | SSG | 필요 시 ISR 검토 가능 |
| 공지 상세 페이지 | `/notice/[id]` | 공개 | SSG 우선 | 실제 갱신 주기 크면 ISR 검토 |
| 공개 포트폴리오 목록 | `/portfolio` | 공개 | SSG | 콘텐츠 규모에 따라 추후 검토 |
| 공개 포트폴리오 상세 | `/portfolio/[id]` | 공개 | SSG 우선 | 현재는 SSR 제외 |
| 사용자 내부 페이지 | `/profile`, `/dashboard` | 비공개 | CSR | 인증/상태/입력 중심 |
| 작성/편집 페이지 | `/portfolio/edit` | 비공개 | CSR | 폼 상태 중심 |

## 3. 도구 사용 기준

### 3.1 폼

- 실제 인증/회원가입 화면 구현 시 `react-hook-form + zod`를 도입한다.
- 모든 폼 화면에 선도입하지 않고, 실제 폼 라우트가 생길 때부터 적용한다.

### 3.2 전역 상태

- `zustand`는 선제 도입하지 않는다.
- 여러 컴포넌트/페이지에서 공유되는 클라이언트 상태가 생길 때만 최소 범위로 도입한다.

예상 후보:

- 모달 열림 상태
- 대시보드 필터 상태
- 다단계 작성 흐름
- 사용자별 UI 설정 상태

## 4. Firebase Hosting 관련 메모

- Firebase Hosting은 `out/` 정적 산출물을 직접 배포한다.
- 현재 구조는 "Next.js 앱 + Firebase Hosting(static export) + SSG 우선 설계"로 이해한다.

## 5. 후속 작업

1. `/about`, `/notice` 등 실제 공개 페이지를 만들고 `SSG` 기준으로 구현한다.
2. 공지사항 데이터 갱신 주기를 보고 `ISR` 필요성을 다시 평가한다.
3. 인증/회원가입 페이지 구현 시 `react-hook-form + zod` 패턴을 적용한다.
4. 공유 클라이언트 상태가 생기면 `zustand`를 최소 범위로 도입한다.
